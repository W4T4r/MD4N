# Private Repo Guide

Use a separate private repository when a machine-local file should persist across rebuilds, stay out of the shared `MD4N` tree, and still be easy to edit in place.

The tracked starter lives in [`../private_templates/MD4N-private`](../private_templates/MD4N-private).

## What Goes Where

- Shared reusable defaults and modules stay in `MD4N/`.
- Generated runtime state stays in `MD4N/local/`.
- Hand-maintained private machine files stay in `MD4N-private/<machine>/`.

Inside each machine directory:

- `local/`
  Versioned local flake files that are synced into `MD4N/local/`.
- `home-manager/`
  Runtime files that are linked into ignored paths under `MD4N/home-manager/config/`.

This split is intentional:

- `local/` is copied into `MD4N/local/` because Nix evaluation is safer against real files than against arbitrary out-of-tree symlinks.
- `home-manager/` files are symlinked so app-managed edits land back in the private repository immediately.

## Bootstrap

Recommended first-time flow:

1. Run the normal setup in `MD4N` once so `local/` and generated helper files exist.
2. Copy `private_templates/MD4N-private/` into a separate sibling repository such as `../MD4N-private/`.
3. Initialize that copied directory as its own Git repository and add your private remote.
4. Run `bash ./new-machine.sh <machine-dir>` inside `MD4N-private`.
5. Run `bash ./link-md4n.sh --machine <machine-dir>` to seed missing files and switch runtime paths to the private-managed copies.

`new-machine.sh` intentionally seeds from tracked templates and shared defaults only.
It does not copy your current `MD4N/local/` private overrides such as machine-specific package wiring.

Example:

```bash
cp -a private_templates/MD4N-private ../MD4N-private
cd ../MD4N-private
git init
bash ./new-machine.sh laptop
bash ./link-md4n.sh --machine laptop
```

## Day-To-Day Management

Treat `MD4N-private` as the source of truth for hand-maintained local files.

- Edit shared logic in `MD4N/`.
- Edit private runtime files in `MD4N-private/<machine>/home-manager/`.
- Re-run `bash ./link-md4n.sh --machine <machine-dir>` when you want to re-apply the private tree to `MD4N/`.

Important direction rules:

- `link-md4n.sh` syncs `MD4N-private/<machine>/local/` into `MD4N/local/`.
- `link-md4n.sh` links `MD4N-private/<machine>/home-manager/` files into `MD4N/home-manager/config/`.
- After linking, editing a linked runtime file through the `MD4N/home-manager/config/...` path edits the private file because that path is a symlink.

## Generated Files

Some files are generated by the shared scripts but still belong in the private repository after linking.

Examples:

- `home-manager/niri/outputs.kdl`
- `home-manager/niri/browser.sh`
- `home-manager/fish/local.env.fish`

After `link-md4n.sh`, those runtime paths inside `MD4N/` point at the private repository, so regenerating them through the normal shared scripts updates the private files in place.

`local/generated/user.nix` is different:

- `scripts/configure-local.sh` generates it inside `MD4N/local/generated/user.nix`.
- `link-md4n.sh` later syncs the private copy back into `MD4N/local/`.

If you re-run `configure-local.sh` and want to keep the new generated answers in your private repository, copy or review the updated `MD4N/local/generated/user.nix` into `MD4N-private/<machine>/local/generated/user.nix` before the next re-link.

## Niri

For Niri, keep machine-only files under `MD4N-private/<machine>/home-manager/niri/`:

- `outputs.kdl`
- `config.local.kdl`
- `local/*.local.kdl`
- `browser.sh`

Shared `config.kdl` loads `config.local.kdl` last, so local overrides win over the tracked shared config.

## Custom Fonts And Fish

The same pattern applies to other reserved local slots:

- `home-manager/custom-fonts/`
- `home-manager/fish/`

Keep the real files in `MD4N-private`, then let `link-md4n.sh` link them into the ignored runtime paths under `MD4N/home-manager/config/`.
